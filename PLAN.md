## 阶段 0：Bug 修复（P0 — 基础）

### 任务 0.1 — 修复引号内标点被错误分割

- **问题**：`normalizeText()` 将引号内的逗号也替换为"然后"，导致 `写上"你好，世界"` 被拆成 `写上"你好 然后 世界"`，文字内容解析错误。
- **文件**：`static/parser.js`
- **改动点**：
  1. 修改 `normalizeText()`：在替换标点前，先提取并保护引号内区域
  2. 修改 `splitCommands()`：只按句末标点（`。！？`）分割，逗号交给逐句解析器处理
- **验收标准**：`parseCommand('画一个文字，写上"你好，世界"')` 正确解析为创建文字动作，内容为 `你好，世界`
- **测试**：`tests/parser.test.js` 新增引号保护测试用例
- **预估**：2h

### 任务 0.2 — 修复确认窗口可被绕过

- **问题**：风险操作触发 8 秒确认窗口期间，说非确认/取消的指令会穿透执行，且 `pendingConfirmation` 状态未清除
- **文件**：`static/app.js`
- **改动点**：
  1. 在 `handleCommand()` 中，当 `pendingConfirmation` 存在时，对所有非 确认/取消 的文本直接忽略
  2. 增加语音提醒："请说确认或取消"
  3. 确保超时后 `pendingConfirmation` 正确清除
- **验收标准**：确认窗口期间说"画一个圆"，画布不变化，`pendingConfirmation` 不受影响
- **测试**：`tests/app.test.js` 新增确认窗口隔离测试用例
- **预估**：1h

### 任务 0.3 — 修复 SVG 导出缺少背景色

- **问题**：背景色仅设置在 CSS `style.background` 上，导出为独立 SVG 文件后背景色丢失
- **文件**：`static/app.js`
- **改动点**：
  1. 在 `download("svg")` 中，当 `engine.state.background` 不是默认白色时，向 SVG 序列化结果插入一个全画布 `<rect>` 作为第一个子元素
  2. 或者修改 `render()` 在 SVG 中始终渲染背景矩形
- **验收标准**：设置红色背景后导出 SVG，用图片查看器打开能看到红色背景
- **测试**：`tests/app.test.js` 新增 SVG 背景测试（检查导出字符串是否包含背景 `<rect>`）
- **预估**：1h

## 阶段 1：免手操作（P1 — 核心体验）

### 任务 1.1 — 页面加载自动恢复聆听

- **目标**：已授权麦克风的用户刷新页面后无需点击任何按钮，自动进入唤醒词监听
- **文件**：`static/app.js`、`static/index.html`
- **改动点**：
  1. 页面加载时立即调用 `navigator.mediaDevices.getUserMedia({ audio: true })`
  2. 成功（权限已授权）→ 自动进入唤醒词监听模式，Toast："正在后台聆听唤醒词'听画'…"
  3. 失败（权限未授权）→ 显示"开始聆听"按钮，引导用户点击授权
- **验收标准**：首次授权后刷新页面，3 秒内自动进入后台监听状态，无需任何点击
- **测试**：`tests/app.test.js` 新增自动启动流程测试
- **预估**：3h

### 任务 1.2 — 唤醒词检测

- **目标**：用户说"听画"/"开始画"/"嘿画布"即可唤醒应用进入完整聆听模式
- **文件**：`static/app.js`
- **方案**：两级检测
  - **第一级（常驻）**：复用现有 VAD 的 `AnalyserNode` 进行能量检测。当 RMS 超过 `SOUND_THRESHOLD` 持续 300ms → 触发第二级
  - **第二级（关键词）**：创建一次短时 `SpeechRecognition`（`continuous: false`，3s 超时），监听唤醒词列表。命中 → 进入完整聆听模式；超时 → 回到第一级
- **新增函数**：
  - `startWakeWordListening()` — 启动两级唤醒词检测
  - `stopWakeWordListening()` — 停止并清理
  - `handleWakeWord(result)` — 检查识别结果是否匹配唤醒词
- **唤醒成功**：TTS 播报"听画已唤醒"，过渡动画，进入 `startCloudListening()`
- **语音休眠**：完整聆听模式下说"休息"/"停止聆听" → 回到唤醒词监听
- **验收标准**：
  1. 后台模式说"听画"→ 进入完整聆听，播报"听画已唤醒"
  2. 完整聆听说"休息"→ 回到后台模式
  3. 后台模式下非唤醒词语音不触发完整聆听
- **测试**：`tests/app.test.js` 新增唤醒词状态机测试
- **预估**：8h

### 任务 1.3 — 语音确认完善

- **目标**：确保风险操作的语音确认不依赖任何点击
- **文件**：`static/app.js`
- **改动点**：
  1. 确认窗口期间暂停常规指令处理（已在 0.2 修复）
  2. 确认窗口期间保持语音识别活跃（只监听"确认"/"取消"/"是"/"否"/"好的"/"不要"）
  3. 超时后自动取消，TTS 播报"已取消"
- **验收标准**：清除画布 → 听到确认提示 → 说"确认"→ 画布清除。整个流程无需点击
- **测试**：`tests/app.test.js` 已有覆盖，补充确认超时测试
- **预估**：2h

### 任务 1.4 — UI 适配免手模式

- **目标**：更新界面文案和状态指示，反映当前模式
- **文件**：`static/index.html`、`static/styles.css`
- **改动点**：
  1. 状态指示器显示三种模式：🌙 后台监听 / 🟢 完整聆听 / 🔴 已停止
  2. 初始按钮文案从"开始聆听"改为"授权麦克风并开始"
  3. 后台监听时显示声波动画（区别于完整聆听）
  4. 唤醒成功时有短暂视觉反馈（画布边框闪烁）
- **验收标准**：三种状态在 UI 上有清晰区分
- **预估**：1h

## 阶段 2：复杂指令拆解（P1 — 智能交互）

### 任务 2.1 — 本地组合图形拆解

- **目标**：无需 LLM 即可识别"画一个房子"等高频语义指令
- **文件**：`static/parser.js`
- **改动点**：
  1. 新增 `decomposeComposite(text)` 函数，在 `parseCommand()` 的最前面调用
  2. 支持以下组合模式：

| 指令模式 | 拆解结果 | 参数计算 |
|---------|---------|---------|
| `画一个房子` | 三角形(屋顶) + 矩形(墙体) + 矩形(门) | 屋顶宽=墙体宽，门居中靠下 |
| `画一个雪人` | 圆(底, r=60) + 圆(中, r=40) + 圆(头, r=25) | 纵向居中堆叠 |
| `画一个笑脸` | 圆(脸) + 圆(左眼) + 圆(右眼) + 椭圆(嘴) | 眼睛在脸上部，嘴在下部 |
| `画一排{N}个{形状}` | N 个 create 动作 | 水平均匀分布，间距=宽×1.5 |
| `画一列{N}个{形状}` | N 个 create 动作 | 垂直均匀分布，间距=高×1.5 |

  3. 复用现有 `chineseNumber()` 解析数量
  4. 复用现有形状关键词匹配
- **验收标准**：
  1. "画一个房子" → 创建 3 个图形（屋顶+墙+门），位置正确
  2. "画一排五个圆" → 创建 5 个水平排列的圆形
- **测试**：`tests/parser.test.js` 新增 5+ 个组合拆解测试用例
- **预估**：6h

### 任务 2.2 — LLM 系统提示词增强

- **目标**：让 LLM 更好地拆解本地未覆盖的语义指令
- **文件**：`main.py`
- **改动点**：
  1. 在 `SYSTEM_PROMPT` 中添加组合拆解示例（房子、树、云朵、星星图案等）
  2. 明确指示 LLM：遇到语义概念时拆解为多个 `create` 动作
  3. 每个动作计算合理的相对位置和尺寸
- **验收标准**：LLM 收到"画一棵树"时返回矩形(树干)+圆形(树冠) 两个 create 动作
- **测试**：`tests/test_server.py` 新增 LLM 拆解验证测试
- **预估**：1h

### 任务 2.3 — 拆解结果的可撤销性

- **目标**：确保组合拆解产生的多个动作作为单一事务执行（一次撤销全部回退）
- **文件**：`static/parser.js`（确认 `parseCommand` 的行为）
- **改动点**：
  1. 确认 `decomposeComposite` 返回的动作数组被 `execute()` 作为一个事务处理（现有机制已支持，验证即可）
  2. 如有需要，在 `model.js` 的事务标记中增加组合指令来源信息
- **验收标准**：执行"画一个房子"后，一次"撤销"移除所有 3 个图形
- **测试**：`tests/model.test.js` 已有事务测试，确认覆盖组合场景
- **预估**：1h

### 任务 3.2 — 识别候选项评分择优

- **目标**：Web Speech API 返回的多个识别候选项中，选择最可能正确的那个
- **文件**：`static/app.js`
- **改动点**：
  1. 重写 `chooseRecognitionAlternative(alternatives)` 函数
  2. 评分规则：
     - 能被 `parseCommand()` 解析 → +10 分
     - 包含已知指令关键词（画/创建/删除/移动/选择/撤销…） → +5 分/个
     - ASR 置信度 → +confidence × 5
     - 文本长度 < 2 字符且无关键词 → -10 分（噪音）
  3. 选最高分候选项；同分选 confidence 高的
- **验收标准**：
  - 候选项 A (confidence 0.9, "嗯") vs 候选项 B (confidence 0.7, "画圆") → 选 B
  - 候选项 A (confidence 0.8, 可解析) vs 候选项 B (confidence 0.9, 不可解析) → 选 A
- **测试**：`tests/app.test.js` 新增候选项评分测试用例
- **预估**：3h

## 阶段 4：延迟优化（P2 — 响应速度）

### 任务 4.1 — 自适应 VAD 静音阈值

- **目标**：减少"用户说完 → 系统开始处理"之间的等待时间
- **文件**：`static/app.js`
- **改动点**：
  1. 参数调整：

| 参数 | 当前值 | 新默认值 |
|------|--------|---------|
| `SILENCE_MS` | 800ms | 500ms |
| `MIN_SPEECH_MS` | 300ms | 250ms |
| `MAX_SEGMENT_MS` | 15000ms | 10000ms |

  2. 自适应逻辑：在 `voiceActivityDecision()` 中根据当前语音段时长动态调整：
     - 语音段 < 2s（短指令）→ 静音阈值 400ms
     - 语音段 ≥ 2s（长指令）→ 静音阈值 600ms
  3. 最小语音段从 300ms 降到 250ms（减少短指令被丢弃的概率）
- **验收标准**：
  1. 短指令"画圆"说完后 ~400ms 开始处理
  2. 长指令"画一个红色的圆然后移动到右边"说完后 ~600ms 开始处理
- **测试**：`tests/app.test.js` 新增自适应 VAD 测试用例
- **预估**：2h

### 任务 4.2 — 中间结果预览渲染

- **目标**：用户说话过程中，Web Speech API 返回的中间结果即时预览，减少体感延迟
- **文件**：`static/app.js`、`static/index.html`、`static/styles.css`
- **改动点**：
  1. `index.html`：在 `<g id="drawing-layer">` 之前添加 `<g id="preview-layer" class="preview">`
  2. `styles.css`：`.preview { opacity: 0.4; }` `.preview * { transition: opacity 0.15s; }`
  3. `app.js`：
     - 在 `recognition.onresult` 中，对非最终的 `interim` 结果调用 `tryPreviewRender(interimText)`
     - `tryPreviewRender()` 尝试解析，成功则渲染到 `#preview-layer`
     - 最终结果到达时，清除 `#preview-layer`，走正式 `handleCommand` 流程
     - 节流：文字变化 < 2 字符或距上次 < 200ms 则跳过
  4. `clearPreview()` 函数清除预览层
- **验收标准**：
  1. 说"画一个红色的圆"时，在"画一个红色"阶段就能看到半透明的圆
  2. 最终结果确认后，预览消失，正式图形以完整不透明度呈现
- **测试**：`tests/app.test.js` 新增预览渲染测试用例
- **预估**：5h

### 任务 4.3 — 正式渲染性能复查

- **目标**：确保新增功能不引入渲染性能退化
- **文件**：`static/app.js`
- **改动点**：
  1. 检查 `render()` 函数中预览层和正式层的切换逻辑
  2. 确保 `clearPreview()` 在每次 `render()` 调用前执行
  3. 测量 100 个图形场景下的渲染耗时，确保 <16ms（60fps）
- **验收标准**：100 个图形的画布渲染 <16ms
- **预估**：1h
