## P0：离线识别闭环（当前最紧迫）

> 现状：离线模式的 UI、模式切换、模型下载进度、IndexedDB 缓存接口、VAD 音频路由和 mock 测试全部已实现。但 `_createRealRecognizer()` 回退到 mock，ZIP 未解压到 Vosk 虚拟文件系统，音频未做 16kHz 重采样。UI 会在模型 ZIP 缓存后显示"模型已就绪"，实际不能产生转写结果。

### 任务 0.1 — 标记离线模式为实验性能力 ✅ 已完成

- **完成日期**：2026-06-13
- **文件**：`static/app.js`、`static/vosk_recognizer.js`
- **实际改动**（超出原计划）：
  1. `updateModeIndicator()`：三级显示 — `"💻 离线"`（真实 WASM 就绪）/ `"💻⚠ 离线(实验性)"`（仅缓存）/ `"💻🚫 离线不可用"`（无模型）
  2. `checkModelAvailability()`：同时检查 `globalThis.VoskModule` 和 IndexedDB 数据，仅两者都就绪才返回 `available: true`
  3. `_initRecognizer()`：VoskModule 不存在时直接设 `_ready=false` 并报告错误，不再走 mock 路径
  4. `createVoskRecognizer()`：暴露 `recognizer.ready` Promise，供调用方准确等待初始化
  5. `initVoskIfNeeded()`：改用 `await voskRecognizer.ready` 替代不稳定的 `setTimeout(50ms)`
  6. `switchVoiceMode("offline")`：不可用时给出明确 toast + 回退到云端/浏览器
  7. `vosk_recognizer.js` 文件头部添加 `⚠️ STATUS` 文档块，说明真实 Vosk WASM 未闭环
- **验收标准**：✅ 模型缓存后 UI 不会暗示离线识别已就绪
- **预估**：1h（实际约 1.5h，含跨文件联动）

### 任务 0.2 — 实现 ZIP 解压与模型虚拟文件系统建立

- **目标**：将下载的模型 ZIP 解压并写入 Vosk WASM 虚拟文件系统
- **文件**：`static/vosk_recognizer.js`
- **改动点**：
  1. 引入 JSZip（或等价 ZIP 解压库），在 `downloadModel()` 的 extract 阶段解压 ZIP
  2. 将解压后的模型文件存储到 IndexedDB（按路径索引，而非单个 ZIP blob）
  3. 实现 `_ensureModelFiles()`：从 IndexedDB 读取模型文件，写入 Vosk WASM 虚拟文件系统
  4. 更新 `checkModelAvailability()`：检查解压后的模型文件是否存在且完整（而非仅检查 ZIP blob）
- **验收标准**：
  1. `downloadModel()` 完成后，IndexedDB 中存储解压后的模型文件
  2. `checkModelAvailability()` 在解压文件存在时返回 `available: true`
  3. 解压文件不存在（仅有旧版 ZIP blob）时返回 `available: false`
- **预估**：8h

### 任务 0.3 — 实现真实 Vosk WASM 识别器初始化与音频重采样

- **目标**：加载真实 Vosk WASM 模块，完成音频到 16kHz 单声道的重采样，实现从语音到文本的离线转写
- **文件**：`static/vosk_recognizer.js`、`static/app.js`
- **改动点**：
  1. 加载 Vosk WASM 模块（通过 `<script>` 标签或动态 import）
  2. `_createRealRecognizer()` 中创建真实的 `KaldiRecognizer`
  3. 在 `feedAudio()` 前实现音频重采样：浏览器默认采样率（通常 44.1kHz/48kHz）→ 16kHz 单声道
  4. 处理 `partialresult` 和 `result` 回调，映射到现有的 `onPartial` 和 `onFinal`
  5. `monitorOfflineAudio()` 中按固定帧长（如 4096 samples @ 16kHz ≈ 256ms）喂给识别器
  6. 离线识别失败时自动降级到云端模式（而非静默失败）
- **验收标准**：
  1. 离线模式下说"画一个圆形"→ 画布上出现圆形
  2. 离线识别失败 → 自动切换到云端模式并提示用户
  3. 即使无网络，基本的"画圆""撤销""移动"等指令可被正确识别
- **预估**：11h（含 Vosk WASM 集成调试和音频链路调试）

## 本轮修复中发现的额外问题（未在原始计划中）✅ 已全部处理

### 修复 A — 移除 SpeechRecognition.phrases 实验性 API

- **问题**：Chrome 实验性 `SpeechRecognition.phrases` API 在 `zh-CN` 上不支持，`recognition.start()` 时抛出 `phrases-not-supported` 错误，浏览器模式完全不可用
- **修复**：删除 `RECOGNITION_PHRASES` 常量（47 条短语）、`addRecognitionPhraseBias()` 函数及其调用点
- **影响文件**：`static/app.js`、`tests/app.test.js`

### 修复 B — 浏览器模式全链路诊断日志

- **问题**：浏览器/云端模式出错时无法定位根因
- **修复**：`handleCommand`、`executeActions`、`transcribeAudio`、`startRecognition`、recognition 事件（`onstart/onend/onerror/onresult`）全部添加 `console.log`
- **影响文件**：`static/app.js`
- **附带**：`static/diagnostic.html` — 浏览器兼容性诊断页面

### 修复 C — `showFallbackPrompt` 浏览器可用性检查

- **问题**：云端失败后切换到浏览器模式时未检查 `recognition` 是否存在（Firefox 无 Web Speech）
- **修复**：`showFallbackPrompt` 分 `recognition` 存在/不存在两条路径
- **影响文件**：`static/app.js`

### 修复 D — `transcribeAudio` API 检测可靠性

- **问题**：原先通过 Content-Type + status 判断是否连接到听画 API，不可靠
- **修复**：改用服务端 `X-Listen-Paint-API: 1` 响应头判断，退化时保留 status+Content-Type 兼容
- **影响文件**：`static/app.js`

### 修复 E — 浏览器 `onerror` 致命错误提示模式切换

- **问题**：浏览器识别致命错误（网络、权限等）后静默停止，不提示可切换云端
- **修复**：fatal error 时 toast 提示可切换云端模式
- **影响文件**：`static/app.js`

### 修复 F — `switchVoiceMode` 离线回退逻辑

- **问题**：切换到离线模式但模型不可用时静默回退
- **修复**：明确 toast 提示原因，回退到云端/浏览器，UI 指示器同步更新
- **影响文件**：`static/app.js`

### 修复 G — 服务端 `_` 前缀字段豁免

- **问题**：`main.py` 的 `validate_action` 拒绝 `_compositeId` 等浏览器侧元数据字段
- **修复**：前后端验证均跳过 `_` 前缀字段
- **影响文件**：`static/model.js`、`main.py`

### 任务 2.6 — LLM 系统提示词增强：覆盖常见修饰语与空间关系

- **目标**：让 LLM 回退路径能正确理解"尖顶的房子""圆顶的房子""大房子""左边的树"等带修饰语的复杂指令
- **背景**：当前 `SYSTEM_PROMPT` 只教了 LLM 标准模板（房子=三角屋顶+矩形墙+矩形门），没有变体示例。实测 LLM 能识别颜色修饰语（"红顶"→fill 生效），但几何修饰语（"尖顶"→屋顶应更窄更高）被忽略。空间关系（"左边有一棵树"）缺乏参照系说明。
- **文件**：`main.py`
- **改动点**：
  1. 在 `SYSTEM_PROMPT` 的拆解示例区域新增变体示例：
     - "画一个尖顶的房子" → 屋顶三角形更窄更高（width 缩小，height 增大），示例坐标
     - "画一个圆顶的房子" → 用半椭圆替代三角形屋顶
     - "画一个矮胖的房子" → 墙体 width 增大，height 缩小
     - "画一个大房子" / "画一个小房子" → 所有组件按比例缩放（大=1.5x，小=0.6x）
  2. 新增空间关系章节：
     - "X的左边有一棵Y" → Y 在 X 的左侧，间距约 50px
     - "X的上方" / "X的下方" / "X的周围"
     - "X和Y之间" → 居中放置
  3. 补充常见语义概念的拆解（不要求全部覆盖，给 LLM 拆解思路）：
     - 城堡：多个矩形塔楼+三角形旗+矩形城墙
     - 花朵：中心圆+周围若干椭圆花瓣
     - 山：多个重叠三角形
     - 月亮：两个圆叠加（一个与背景同色做裁切效果）
  4. 强调画布尺寸 1000×700、中心 (500,350)，所有坐标计算以此为参照
- **验收标准**：
  1. "画一个尖顶的房子" → 屋顶三角形 height > 标准值，width < 标准值
  2. "画一个圆顶的房子" → 屋顶为椭圆（kind: "ellipse"）而非三角形
  3. "画一个红顶的房子，左边有一棵小树" → 树在房子的左侧（x 坐标小于房子 x 坐标）
  4. "画一个城堡" → ≥4 个动作，包含塔楼和城墙
- **测试**：`tests/test_server.py` 新增 SYSTEM_PROMPT 覆盖率测试（提示词包含"尖顶""圆顶""城堡"等关键词）
- **预估**：2h
- **优先级说明**：这是复杂指令的**唯一处理路径**。本地规则穷举不了修饰语组合，LLM 回退是结构化兜底。提示词质量直接决定系统能理解多复杂的指令。

### 任务 2.7 — 本地修饰语剥离：离线模式下也能处理修饰复合指令

- **目标**：在 `decomposeComposite` 匹配前，剥离已知修饰语（颜色、尺寸形容词），使"画一个红色的尖顶大房子"能退化为"画一个房子"命中本地模板。修饰语信息尽可能保留并作用到子动作上。
- **背景**：P0 闭环后，离线模式不能依赖 LLM。当前复合模板要求整句精确匹配（如 `/^(?:画(?:一个)?(?:房子|房屋))\s*$/`），任何修饰语都会导致匹配失败。需要在匹配前做归一化。
- **文件**：`static/parser.js`
- **改动点**：
  1. 新增 `stripModifiers(text)` 函数：
     - 剥离颜色词：`/红色|蓝色|绿色|黄色|.../` → 提取到 `modifiers.fill`
     - 剥离尺寸词：`/大的|小的|巨大的|微小的/` → 提取到 `modifiers.scale`
     - 剥离形状词：`/尖顶的|圆顶的|矮胖的|细长的/` → 提取到 `modifiers.shape`
     - 剥离位置词：`/左边的|右边的|上方的|下方的/` → 保留给后续 parseClause
     - 返回 `{ stripped: "画一个房子", modifiers: { fill: "红色", scale: "大", shape: "尖顶" } }`
  2. `decomposeComposite` 匹配时先调用 `stripModifiers`，用剥离后的文本匹配模板
  3. 匹配成功后，将提取的修饰语应用到子动作：
     - `fill` → 应用到顶层组件（如房子的屋顶）
     - `scale` → 所有子动作的 width/height 乘以系数
     - `shape` → 触发几何参数调整（尖顶→屋顶三角形更窄更高）
  4. 剥离失败（不认识的修饰语）→ 保留原文本，走 LLM 回退
- **验收标准**：
  1. "画一个红色的大房子" → 命中本地房子模板，屋顶 fill=#ef4444，所有组件尺寸 ×1.5
  2. "画一个尖顶的房子" → 命中本地房子模板，屋顶三角形 height 增大、width 缩小
  3. "画一个xyz房子" → 剥离失败，走 LLM 回退（不丢数据）
- **测试**：`tests/parser.test.js` 新增 `stripModifiers` 单元测试
- **预估**：3h

## P3：组合与工程格式

> 现状：组合只是成员关系元数据，不是一等对象；刷新页面丢失所有状态，没有保存/加载能力。

### 任务 3.1 — 组合升级为一等可命名对象

- **目标**：组合有自己的名称、可以被整体选择，操作组合时自动作用于所有成员
- **文件**：`static/model.js`、`static/parser.js`、`static/app.js`
- **改动点**：
  1. `model.js`：
     - 组合创建时生成名称（如"组合一""组合二"）
     - 支持 `select target: "组合一"` 展开为组合内所有成员
     - 组合成员变更时更新组合的包围盒信息
  2. `parser.js`：
     - 支持"选择组合一""把组合一向右移动"等指令
     - "组合一"作为 `target` 解析为组合成员 ID 数组
  3. `app.js`：
     - 渲染时对选中的组合显示组合边框
- **验收标准**：
  1. 选中 3 个图形 → "组合" → 生成"组合一"
  2. "选择组合一" → 3 个图形全部选中
  3. "把组合一向右移动五十" → 3 个图形同时移动
- **测试**：`tests/model.test.js` 新增组合名称和整体操作测试
- **预估**：6h

### 任务 3.2 — 工程文件保存与加载（JSON 格式）

- **目标**：将完整画布状态（图形、历史、组合、背景）序列化为 JSON，支持保存到本地文件和加载恢复
- **文件**：`static/model.js`、`static/parser.js`、`static/app.js`
- **改动点**：
  1. `model.js`：
     - 新增 `serialize()` 返回完整状态 JSON（含 undoStack、groups、counters）
     - 新增 `deserialize(json)` 恢复完整状态
  2. `parser.js`：
     - 新增"保存工程""导出工程""加载工程""打开工程"指令
  3. `app.js`：
     - "保存工程"→ 序列化为 JSON → 触发下载为 `.listenpaint.json`
     - "加载工程"→ 语音提示用户点击上传（浏览器安全限制文件读取需用户手势）
     - 页面加载时可选地从 `localStorage` 自动恢复上次状态
  4. 自动保存到 `localStorage`：每次操作后自动保存（防崩溃丢失）
- **验收标准**：
  1. 画布上有图形 → "保存工程" → 下载 `.listenpaint.json` 文件
  2. 刷新页面 → 画布自动恢复到刷新前状态（localStorage）
  3. 手动加载 `.listenpaint.json` 文件 → 画布恢复
- **测试**：`tests/model.test.js` 新增序列化/反序列化测试
- **预估**：5h

### 任务 3.3 — 崩溃恢复（自动保存）

- **目标**：页面意外关闭或崩溃后，下次打开时能恢复到最近状态
- **文件**：`static/app.js`
- **改动点**：
  1. 使用 `beforeunload` 事件 + 定时自动保存到 `localStorage`
  2. 页面加载时检测 `localStorage` 中是否有未恢复的工程
  3. 如有 → TTS 播报"检测到上次未保存的工程，已自动恢复"
  4. 用户可选择"丢弃上次工程"从空白开始
- **验收标准**：
  1. 画布上有图形 → 关闭标签页 → 重新打开 → 图形仍在
  2. 用户说"丢弃上次工程"→ 清空画布
- **预估**：1h

### 任务 4.1 — 建立语音验收检查表与执行记录

  4. 唤醒词检测准确率和误触发率（安静/嘈杂环境各 10 分钟）

- **验收标准**：
  1. 云端模式：20 条标准指令准确率 ≥ 90%
  2. 浏览器模式：20 条标准指令准确率 ≥ 70%
  3. 简单指令端到端延迟 < 1s
- **预估**：3h

### 任务 4.3 — 唤醒词长期稳定性验收

  1. 安静环境误触发率：0 次/30 分钟
