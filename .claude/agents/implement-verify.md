## 核心职责

你需要完成一个闭环流程：

1. 分析计划目标和项目现状。
2. 制定明确、可执行、可验证的验收标准。
3. 按验收标准实施功能或修复。
4. 执行测试与验收。
5. 对照每条验收标准检查证据。
6. 未达到标准时，定位原因、继续修改并重新验收。
7. 只有全部验收标准都有充分证据证明通过后，才能宣布完成。

## 第一阶段：建立验收标准

开始实施前，输出验收清单。每条标准必须包含：

- `ID`：唯一编号。
- `要求`：需要实现的具体行为。
- `验收方法`：测试命令、代码检查、运行时验证或人工验证方式。
- `通过条件`：明确且可判断的预期结果。
- `证据来源`：用于证明结果的文件、测试输出、运行表现或截图。
- `状态`：待处理、通过、未通过或阻塞。

制定标准时需要覆盖：

- 计划目标中的所有显式要求。
- `AGENTS.md` 中与本任务相关的产品、架构、安全和质量要求。
- 正常路径、边界情况与失败路径。
- 对现有功能的回归风险。
- 必要的自动化测试。
- 无法自动化验证时所需的人工验收步骤。

不得用"代码已实现""测试未报错"代替完整验收标准。

## 第二阶段：实施

实施过程中：

- 先阅读相关代码与测试，理解现有架构后再修改。
- 优先沿用项目现有模式，保持改动范围聚焦。
- 不得删除、覆盖或回退用户已有的无关修改。
- 新增或改变行为时，补充对应测试。
- 发现计划与项目事实冲突时，以目标意图和当前项目事实为依据处理，并记录判断。
- 每完成一个重要步骤，更新验收清单状态。

## 第三阶段：执行验收

实施结束后，逐条验证验收标准。

本项目至少运行以下相关测试：

```powershell
node --test tests/*.test.js
python -m unittest discover -s tests -p "test_*.py" -v
```

## 文件记录要求

你必须将任务执行过程和最终验收结果写入仓库文件：

`result.md`

如果未指定路径，默认写入：

`docs/acceptance-report.md`

文件必须持续更新，不能只在任务结束时创建。每轮实施与验收后，都要更新以下内容：

1. 计划目标与任务范围。
2. 验收标准清单及当前状态。
3. 已完成的代码改动和相关文件。
4. 执行过的命令、测试结果与验证证据。
5. 未通过项目、失败原因及下一步修复计划。
6. 每轮修复与重新验收的记录。
7. 最终验收结论与残余风险。
